## Qué es y por qué importa

Azure Fundamentals es una serie de 3 rutas de aprendizaje. La primera es sobre computo, networking y servicios de storage; la segunta ruta es sobre mejores practicas de seguridad cloud y la ultima sobre govierno y manejo.

Esta certificacion es para personas que tienen interes en Azure o computacion en nube.

No se requiere experiencia y un conocimiento general en TI ayuda con el aprendizaje.

---

## Cómo funciona

La certificacion evalua 4 Dominios:

### [Conceptos cloud - 25-30%]

- **Computacion en nube:** Entrega de servicios de computo (VM, storage, bases de datos y redes) sobre internet. Los serivicios en nube tambien se expanden a IoT, ML y AI. Como esta construido sobre internet, no es necesario esperar a construir un nuevo data center para crecer rapidamente.

- **Modelo de responsabilidad compartida:** Seguridad fisica, energia, refrigeracion y las redes (fisicas) son responsabilidad del CSP (cloud service provides), mientras que el consumidor es responsable por la informacion y datos que estan almacenados en la nube; el consumidor es responsable por el acceso y seguridad de esta informacion.

- **Modelos cloud:** Nube privada, Nube publica (Azure), Nube Hirbida (conexion entre privada y publica) y multicloud (varios CSP como Azure o AWS). Azure Arc ayuda a manejar modelos publicos, hibridos y multicloud.

- **Metodos de precio:** Pay as you go o modelo de pago por uso. En vez de invertir en Infraestructura inicial (CapEx) se paga a medida que se va usando los servicios (OpEx).

- **Alta disponibilidad:** Va de la mano con los SLAs (Service Level Agreement) definidos, un mayor SLA incrementa el costo. 99% de disponibilidad es diferente que 99.9% de disponibilidad.

- **Escalamiento:** El escalamiento vertical añade recursos como RAM y CPU a la maquina. El escalamiento horizontal añade mas instancias.

- **Confiabilidad:** Habilidad del sistema de recuperarse ante fallos. Es un pillar del Well-Architected Framework.

- **Predecibilidad:** Predecir costos y rendimiento.

- **Sostenibilidad:** Ajustar el tamaño (Conincidir recursos con demanda), automatizar (Desescalar y desasignar recursos sin uso), optimizar (Escoger los servicios y configuracion) y monitorear (Uso y gasto)

- **Infraestructura fisica:** 

    1. **Geografia:** Residencia de los datos y limites de cumplimiento (USA, EU, etc).

    1. **Region:** Uno o mas datacenter conectados con baja latencia (East US, West Europe, etc). Existen pares entre regiones (West US empareja con East US) ubicadas a 300 millas para la replicacion de informacion. Existen regiones "soberanas" que son instancias de Azure aisladas de la instancia principal con el fin de propositos legales. 

    1. **Zona de disponibilidad:** Datacenter separados fisicamente (AZ 1, AZ 2, etc). Pueden haber costos de transferencia de datos entre zonas de disponibilidad.

    1. **Datacenter:** Servidores con alimentacion, refrigeracion y red dedicados. 


### [Arquitecturas y servicios en Azure - 35-40%]

Azure cuenta con una jerarquia para la division de cargas de trabajo en la nube:

1. **Cuenta de Azure:** Es

1. **Grupo de gestion:** Es un grupo que contiene varias suscripciones. organiza las suscripciones con el fin de dar mejores condiciones de gobernanza. Pueden ser anidados 6 niveles (sin contar el root ni la suscripcion).

1. **Suscripcion:** Es una unidad de organizacion. Separa el billing y el acceso. Contiene 1 o varios RGs. Cada suscripcion genera una factura separada.

1. **Grupo de recursos:** Eliminar un grupo elimina TODOS los recursos. Un RG no puede contener otro RG.

1. **Recursos:** Es el bloque basico de Azure. Todo lo que se crea, provisiona o despliega es un recurso. Cada recurso pertenece a solo 1 RG. Los recursos se pueden mover de RG.


Servicios de computo:

1. **Azure Virtual Machines (VMs):**
    - Servicio para usar servidores virtualizados como IaaS.
    - Son buenas opciones para tener todo el control del SO, correr software custom y usar configuraciones de hosting custom.
    - Puede usarse como una extension del datacenter on-premise, como DR o con una migracion tipo Lift and shift (rehosting).
    - Poseen un tamaño (proposito, numero de cpus y cantidad de RAM), un disco de almacenamiento (HDD, SDD, etc) y configuracion de red (IPs pub y priv, puertos y subredes).
    - Se divide por familias:
        - B: Burstable                      -> Costo eficiente
        - D: Disk/General                   -> Cualquier uso
        - E: Enterprise / Extra Memory      -> Bases de datos en memoria o trabajos analitica
        - F: Fast Compute                   -> Trabajos con alto uso de CPU
        - L: Local Storage / Low Latency    -> Alto rendimiento en procesamiento de datos
        - M: Massive Memory                 -> Memoria en orden de terabytes
        - N: NVIDIA / GPUs                  -> Trabajos con IA o inferencia, trabajo con graficos
    - Nombre: Standard_D2s_v5 -> Dice que es la familia D, tiene 2 cpus, soporta disco premium y es de la generacion v5.
    - Los Availability Sets protegen la aplicacion contra fallos de hardware (estatico -> Uno crea las VMs manualente).
    - Scale Sets ajustan las VMs de acuerdo a la cantidad de maquinas qeu se necesiten en el ademanda (dinamico).

1. **Azure virtual desktop:**
    - Servicio para usar escritorios virtuales (Usa como base una VM).
    - Permite la integracion con Entra ID.
    - Utiles para personas que necesitan las mismas herramientas pero se dividen el trabajo en turnos.
    - Ayuda a reducir la data expuesta en los dispositivos locales.

1. **Azure containers:**
    - Ejecucion de contenedores.
    - Usado para arquitecturas de microservicios
    - Azure Container Instances:
        - El camino mas rapido y simple para ejecutar contenedores en AZure.
        - Es un PaaS.
    - Azure Container Apps:
        - Incluye un balanceador y scaling.
        - Es un PaaS.
    - Azure Kubernetes Service:
        - Servicio de orquestacion de contenedores.
        - Manejo de l ciclo de vida de los contenedores.

1. **Azure functions:**
    - Es computo serverles.
    - Es event-driven (basado en eventos).
    - Son stateless por defecto pero puede ser statefull.

1. **Azure AI services:**
    - Servicios para lenguaje, habla, vision y procesamiento de documentos.
    - Util para agregar features atraves de una API en vez de entrenar un modelo propio.
    - Azure OpenAI Service ayuda a soportar IA generativa.

1. **Agentic AI patterns:**
    - Combina modelos de IA con instrucciones, contexto y herramientas para compretar una tarea que tiene varios pasos.
    - Los patrones se combinan con servicios de Azure AI y con la logica de la app.

1. **Azure Machine Learning:**
    - Usado cuando se necesita contruir, entrenar y manejar un modelo propio.
    - Se adapta mejor a escenarios donde hay que desarrollar, experimentar y gestionar el ciclo de vida del modelo.

1. **IoT and Edge services**
    - Manejo de dispositivos.
    - Azure IoT Hub: Habilitar seguridad, comunicacion bidireccional entre cloud y el dispositivo.
    - Azure IoT Central: Plataforma SaaS para desarrolladores de soluciones.
    - zure IoT Edge: EXtiende el cloud en los dispositivos para que las cargas se ejecuten donde se generan los datos.


Servicios de almacenamiento:

1. **Blobs:**
    - Servicio de objetos para texto, contenido multimedia, logs o datos binarios.
    - Soporta analisis de big data a traves de Data Lake Storage.
    - Utiles para contenido servido en browser, backups, archive y recuperacion de desastres.
    - Se accede por HTTP/S.
    - Tiene tiers: hot (acceso frecuente), cool (datos no frecuentes que son almacenados por lo menos 30 dias), cold (datos infrecuentes que son almacenados por lo menos 90 dias) y archive (datos que son rara mente accedidos y son almacenados por lo menos 180 dias)

1. **Files:**
    - Servicio de archivos compartidos en cloud o despliegues en on-premise.
    - Se acceden por SMB o NFS.

1. **Queues:**
    - Servicio de "mensajeria" para la comunicacion entre componentes de aplicaciones.
    - Es asincrono.
    - Se accede por HTTP/S.
    - Soporta mensajes de 64 kb.

1. **Tables:**
    - Servicio de NoSQL para datos no relacionales.


Redundancia en almacenamiento:

1. **LRS:**
    - Locally redundant storage.
    - 11 nueves.

1. **ZRS:**
    - Zone-redundant storage.
    - 12 nueves.

1. **GRS:**
    - Geo-redundant storage.
    - 16 nueves.

1. **GZRS:**
    - Geo-zone-redundant storage.
    - 16 nueves.

1. **RA-GRS:**
    - Read-access geo-redundant storage.
    - 16 nueves.

1. **RA-GZRS:**
    - Read-access geo-zone-redundant storage.
    - 16 nueves.

**Nota:** Los nueves son por año, la replicacion en la region secundaria es solo lectura, el RPO entre regiones pares es de 15 minutos y si se replica data de 3 zonas de la region principal la region par tendra 3 copias en una sola zona.


Tipos de almacenamiento:

1. **Standard general-purpose v2:**
    - Recomendado para la mayoria de escenarios.
    - Soporta Blobs (incluyendo Data Lake Storage), Queues, Tables y Files.
    - Redundancia LRS, GRS, RA-GRS, ZRS, GZRS y RA-GZRS

1. **Premium block blobs:**
    - Recomendado para escenarios de alta tasa de transacciones que usan objetos pequeños o que requieren una latencia baja y constante.
    - Soporta Blobs (incluyendo Data Lake Storage)
    - Redundancia LRS, ZRS.

1. **Premium file shares:**
    - Recomendado para aplicaciones de alto rendimiento o gran escala que requieren soporte para SMB o NFS.
    - Soporta Files.
    - Redundancia LRS, ZRS.

1. **Premium page blobs:**
    - Recomendado para cargas de trabajo de paginas, como por ejemplo un disco de una VM.
    - Soporta Page Blobs.
    - Redundancia LRS.

Endpoints de cuentas de almacenamiento

- Los nombres de las cuentas de almacenamiento son unicos, siguiendo las siguientes reglas:
    1. Los nombres deben tener entre 3 a 24 caracteres de longitud y solo puede contener numeros y minusculas.
    1. Deben ser unicos en Azure.

- Los endpoints siguen la siguiente estructura:
    - Blob Storage:
    	- NOMBRE.blob.core.windows.net
    - Data Lake Storage Gen2:
        - NOMBRE.dfs.core.windows.net
    - Azure Files:
    	- NOMBRE.file.core.windows.net
    - Queue Storage:
    	- NOMBRE.queue.core.windows.net
    - Table Storage:
    	- NOMBRE.table.core.windows.net

Migracion de datos en Azure:

1. **Azure Migrate:**
    - Migracion de datos/cargas de on-premise a cloud.
    - Plataforma unificada para empezar, correr y seguir una migracion a Azure.
    - Incluye Azure Migrate: Discovery and assessment y Azure Migrate: Server Migration.
    - Se puede integrar con otros servicios y ISVs(independent software vendor).

1. **Azure Data Box:**
    - Es un servicio de migracion fisico que ayuda a transferir grandes cantidades de datos.
    - Capacidad de 80 TB.
    - El Data Box se transporta desde y hacia un centro de datos a través de un transportista regional y para la proteccion durante el transporte se usa una carcasa que protege el Data Box.
    - Una vez se importen los datos a Azure el disco es borrado de acuerdo al estandar NIST 800-88r1.

1. **AzCopy:**
    - Es una utilidad de línea de comandos.
    - Permite copiar archivos desde y hacia una cuenta de almacenamiento.
    - Se pueden cargar, descargar archivos, sincronizar archivos y copiarlos entre cuentas de almacenamiento.
    - Puede usarse para facilitar la transferencia de archivos entre diferentes nubes.
    - Es unidireccional.

1. **Azure Storage Explorer:**
    - Es una  aplicacion grafica para manejar archivos y blobs en cuentas de almacenamiento.
    - Usa AzCopy en el backend para hacer todas las tareas.

1. **Azure File Sync:**
    - Centraliza los archivos y mantiene la flexibilidad, rendimiento y compatibilidad con Windows file server.
    - Se descarga en un servidor de Windows.
    - Maneja protocolos como SMB, NFS y FTP.
    - Maneja cache con los archivos mas frecuentes, a demas, los archivos que no se acceden con tanta frecuencia se pueden mantener en nube en lo que se solicita.
    - Puede reemplazar un servidor de almacenamiento averiado.
    - Es bidireccional.


Identidad, acceso y seguridad

- **Azure directory services:**
    - relacionado con el modulo de entraid*
    - Microsoft Entra ID
    - Microsoft Entra Connect
    - Microsoft Entra Domain Services
        - ADS en cloud sin tener que gestionar un controlador de dominio.
        - Despliega 2 windows servers en la region seleccionada, pero Azure maneja todo, backups, encripcion, el dominio, etc.
        - La sincronizacion con este servicio y entra ID es unidireccional, o sea que entra ID sincroniza a los servidores de Entra Domain Service y no viceversa como pasa con Entra Connect (conexion al on-prem).

- **Azure authentication methods:**
    - single sign-on (SSO)
        - Ataca la proliferación de contraseñas (multiplicacion o esparcimiento de estas, se unifica en una sola).
    - multifactor authentication (MFA)
        - Añade una capa extra de factores para la authenticacion, si un atacante obtiene la contraseña necesita este segundo facto para lograr un login correcto.
        - Tiene 3 categorias, user knows (preguntas), user has (dispositivos) y user is (biometricos).
    - Microsoft Entra multifactor authentication
        - Servicio de Azure que añade mas opciones al MFA tradicional, como llamadas o notificaciones mobiles.
    - Passwordless authentication
        - Reemplazan las contraseñas confiando en dispositivos + biometricos o PINs.
        - Entra ID soporta Windows Hello for Business, aplicacion Microsoft Authenticator y llaves de seguridad FIDO2.

- **Microsoft Entra External ID:**
    - Una indentidad extena es una persona, dispositivo o servicio que existe fuera del tenant.
    - Ayuda a invitar a terceros sin tener que administrar credenciales locales por cada invitado.
    - No es SSO es cross-tenant y los externos pueden traer sus propias credenciales.
    - Capacidades:
        - B2B collaboration: Colaboracion con usuarios externos por medio de sus identidades preferidas. Los usuarios se representan como usuarios inivitados en el directorio.
        - B2B direct connect: Conexion bidireccional de confianza con otra tenant. Admite canales compartinos de Teams. Los ususaiors no aparecen en el directorio pero son visibles desde el canal compartido de Teams.
        - Microsoft Entra External ID for customers (Azure AD B2C): Identidad de acceso por medio de SaaS o software a medida para consumidores usando Entra External ID para administrar las identidades y accesos.
